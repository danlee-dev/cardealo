# Lessons Learned: Software Process and Management

## 1. 버전 관리 전략

### 1.1 Git Flow 브랜칭 전략
본 프로젝트는 Git Flow 기반의 브랜칭 전략을 채택하여 체계적인 코드 관리를 수행하였다.

- **main**: 배포 가능한 안정화된 코드만 유지. 프로덕션 환경에 직접 배포되는 브랜치
- **develop**: 개발 통합 브랜치. 각 기능 개발이 완료되면 이 브랜치로 병합
- **feature/***: 개별 기능 개발을 위한 브랜치. develop에서 분기하여 기능 완성 후 develop으로 병합

실제 개발 과정에서 `feature/corporate-card`, `feature/payment-system`, `feature/admin-dashboard` 등의 feature 브랜치를 생성하여 각 기능을 독립적으로 개발하였다. 이를 통해 팀원 간 작업 충돌을 최소화하고, 기능별 코드 리뷰가 용이해졌다.

### 1.2 GitHub Issues 활용
프로젝트 전반에 걸쳐 GitHub Issues를 적극 활용하여 작업을 관리하였다.

- **버그 리포트**: 발견된 버그를 이슈로 등록하고 우선순위를 설정
- **기능 요청**: 새로운 기능 요구사항을 이슈로 문서화
- **작업 추적**: 각 이슈에 담당자를 할당하고 진행 상황을 코멘트로 업데이트
- **브랜치 연동**: feature 브랜치 생성 시 관련 이슈 번호를 참조 (예: `feature/123-payment-fix`)

이슈 기반 개발을 통해 작업의 투명성이 확보되었고, 프로젝트 히스토리 추적이 용이해졌다.

### 1.3 커밋 컨벤션
Conventional Commits 규칙을 준수하여 일관된 커밋 메시지를 작성하였다.

```
feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 수정
style: 코드 포맷팅
refactor: 리팩토링
test: 테스트 코드
chore: 빌드 설정 등 기타 변경
```

## 2. CI/CD 파이프라인

### 2.1 Railway 기반 자동 배포
Railway 플랫폼을 활용하여 지속적 배포 환경을 구축하였다.

- main 브랜치에 병합 시 자동 배포 트리거
- 환경별 설정 분리 (개발/스테이징/프로덕션)
- 배포 로그 및 롤백 기능 활용

### 2.2 데이터베이스 마이그레이션
Alembic(FastAPI)과 Flask-Migrate(Flask)를 사용하여 DB 스키마 변경을 버전 관리하였다.

```
alembic upgrade head    # FastAPI (admin-backend)
flask db upgrade        # Flask (user-backend)
```

## 3. 개발 프로세스에서 배운 점

### 3.1 성공 요인
- **명확한 브랜칭 전략**: feature 브랜치 사용으로 병렬 개발이 가능했음
- **이슈 기반 개발**: GitHub Issues를 통한 체계적인 작업 관리
- **자동화된 배포**: 수동 배포 오류를 방지하고 배포 시간 단축
- **코드 리뷰 문화**: PR을 통한 코드 품질 유지

### 3.2 개선이 필요한 부분
- 초기 브랜치 전략 미흡으로 일부 충돌 발생
- 이슈 템플릿 미사용으로 이슈 품질 불균일
- 테스트 자동화 미비로 수동 테스트에 의존

### 3.3 향후 적용할 점
- 이슈 및 PR 템플릿 도입
- GitHub Actions를 활용한 CI 파이프라인 구축
- 단위 테스트 및 통합 테스트 자동화
- 코드 커버리지 측정 도입

## 4. 결론

본 프로젝트를 통해 체계적인 버전 관리와 협업 프로세스의 중요성을 체감하였다. Git Flow 브랜칭 전략과 GitHub Issues를 활용한 이슈 기반 개발은 프로젝트의 가시성과 추적성을 높여주었다. 특히 feature 브랜치를 통한 기능별 개발과 develop 브랜치에서의 통합 과정은 코드 품질을 유지하면서도 빠른 개발 속도를 가능하게 하였다. 향후 프로젝트에서는 CI 파이프라인 강화와 테스트 자동화를 통해 더욱 견고한 개발 프로세스를 구축할 계획이다.
